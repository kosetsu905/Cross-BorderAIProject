4. Production: Replace asyncio.create_task with a Task Queue
Current Issue:
asyncio.create_task(self._run_job(...)) runs the job inside the FastAPI worker. If a CrewAI job takes 5 minutes (which they often do), it ties up that worker. If you get 10 requests simultaneously, your server might crash or time out.
Improvement:
Use Celery or RQ (Redis Queue).
FastAPI pushes the job to Redis.
A separate "Worker" process picks up the job and runs the Crew.
This separates your "API Latency" from "AI Processing Time."

6. Smart Retries in celery_worker/tasks.py

Current Issue:
You are using self.retry(exc=e, countdown=60) inside a try/except block that catches all exceptions.

#Existing code

except Exception as e:
    logger.warning(f"Marketing task retrying... Error: {e}")
    raise self.retry(exc=e, countdown=60)


Risk: If a task fails because of a logic error (e.g., invalid JSON input, CrewAI hallucination loop, or missing environment variable), Celery will retry it 3 times, wasting tokens and money, before finally failing.

Improvement:
Distinguish between Transient Errors (Network/Rate Limit) and Fatal Errors (Logic/Validation).

#New code 

from celery.exceptions import Retry

try:
    return run_marketing_crew(inputs)
except Exception as e:
    # Check if it's a retryable API error
    if "RateLimitError" in str(e) or "APIConnectionError" in str(e):
        raise self.retry(exc=e, countdown=60)
    else:
        # Fail immediately if it's a logic error to save tokens
        logger.error(f"Task failed fatally: {e}")
        raise e
